# Business Case

## Problem

Telecom restoration/provisioning cases are often marked resolved in business systems before service is actually confirmed live. This causes repeat contacts, SLA breaches, churn risk, service credits, manual rework, and weak auditability.

## Buyer / Operator

Likely owners:

- telecom operations,
- service assurance,
- customer support operations,
- network operations,
- digital transformation / automation COE,
- compliance/audit stakeholders.

## Value Metrics

Primary:

- MTTR reduction.
- SLA breach reduction.
- wrongful closure reduction.
- repeat-contact reduction.
- audit completeness.

Secondary:

- lower manual handoff count,
- faster route to correct remediation team,
- lower service-credit exposure,
- fewer reopened cases,
- better agent calibration and usefulness.

## Why UiPath

UiPath is credible because the workflow needs:

- long-running case management,
- human-in-loop decisioning,
- agents for text interpretation,
- APIs/robots for legacy system lookup/remediation,
- process state and auditability,
- eval/regression checks for AI-infused workflow behavior,
- governed improvement loop.

## Differentiation From UiPath One NZ Case Study

One NZ validates telecom orchestration and provisioning acceleration. This project must not merely imitate acceleration. The wedge is:

- safe closure under missing/stale/contradicting evidence,
- agent recommendation visibly overridden by policy,
- adversarial customer/support pressure handling,
- structured evidence packet,
- eval-backed agent usefulness,
- governed policy improvement.
