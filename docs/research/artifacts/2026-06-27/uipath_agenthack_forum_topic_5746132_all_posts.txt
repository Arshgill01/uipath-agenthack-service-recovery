 Track Selection 
 Q: For a solution that includes both structured BPMN automation and Maestro Case escalation — should it be submitted under Track 1 (Maestro Case) or Track 2 (Maestro BPMN)? → You can choose the track that covers best your solution.
 
 Submission & Judging 
 Q: What does the finale consist of, do you rebuild your solution live in front of judges, or present it? Finalists present their working solution live on Zoom to a judging panel and take questions. You are not expected to rebuild anything, it’s a presentation and demo of what you’ve already built. Finalists can make changes in their solutions between 15th of June and 22nd of July.
 Q: Who retains the IP rights to submitted code, the developer or UiPath? You retain full IP rights to your submission. UiPath receives only a non-exclusive license to use it for judging and promotional purposes. One thing to keep in mind: submissions require a public GitHub repo with an open-source license (MIT or Apache 2.0), so the version you submit will be publicly visible. Nothing in the rules prevents you from deploying a separate proprietary version in a real-world context after the hackathon.
 Q: How do coding agents (e.g. Claude Code) need to be demonstrated to qualify for bonus points, in the demo video, the GitHub repo, or both? To qualify for bonus points, you need to document in your Devpost project description or GitHub README: (a) which coding agent tool was used, (b) how it contributed to the solution, and (c) at least one form of verifiable evidence — such as a prompt log, screenshots of the interaction, or a dedicated README section. The more thorough your documentation, the better your chances of receiving the full 2-point bonus.
 Technical — Maestro 
 Q: Is there a recommended way to handle retry logic for failed steps within a Maestro BPMN flow? The execution trail will show a “Retried” badge, so screenshots and video will help document this. You can also configure retries directly, see the element-level retries section in the Maestro properties panel docs .
 Q: I want to use a Gmail event trigger to start a Maestro Case process, but only for emails related to a specific use case (e.g. loan applications) — how do I avoid unnecessary instances being created? You can add filters to Maestro triggers directly from the properties panel — this should handle most filtering without needing a separate RPA step. If the subject line is unpredictable but the content is contextually relevant, one option is to use an RPA step upstream to evaluate the email semantically before invoking Maestro.
 Q: Where can we find documentation for Maestro Case and Coded Apps? Coded Apps is now GA. Here are the relevant docs:

Coding agents are good at generating code, but what they can’t do is manage what happens in production: retries when an API fails, audit trails for compliance, orchestration across SAP, Salesforce, and a legacy desktop app simultaneously. That’s what UiPath provides. The coding agent writes the script and UiPath makes it run at enterprise scale.

--- POST 72 @ForsakenFiji 2026-06-22T13:52:30.507Z ---
This almost certainly isn’t a serverless/licensing problem on your end, your setup (platform units, unattended runtime, Default Serverless assigned to the folder) sounds correct. The tell is “0 elements executed.” That means the case instance was created but the orchestrator never fired the first element, so it’s not “no robot picked it up,” it’s “nothing told the case to start walking.” Serverless robots only get consumed once you reach an agent/process task inside a stage; you never get that far, so “0 sessions” is a symptom, not the cause.
 What bit me with the exact same symptoms: a missing mainline start event in the compiled case definition. A Maestro Case needs a bpmn:startEvent (a message start, it carries a bpmn:messageEventDefinition/ ) in the compiled caseplan.json.bpmn, wired by a sequence flow into your first node (the initial-variables/setup node). That start event is what the case-entered entry condition fires against to kick off stage 1. If it’s absent, you get exactly what you’re describing:
 
 job goes Running with RuntimeType: CaseManagement, no host, no incident, no error, 0 elements, sits forever.
 
 Why it goes missing: the Studio Web canvas tends to drop the mainline start event when you round-trip the case (edit → re-save/download → re-pack).
 How to confirm/fix:
 
 Open the compiled caseplan.json.bpmn (not caseplan.json) and check there’s a <bpmn:startEvent …> with a bpmn:messageEventDefinition/ and an outgoing bpmn:sequenceFlow into your first node. If it’s not there, that’s it.
 If you authored/edited in the canvas, re-verify it survived the last save, re-add the message start event and the flow into the first node, then re-pack/redeploy. (I keep a tiny
merge script that re-applies the start event after every canvas download precisely because the canvas keeps dropping it.)
 Sanity note: a bare jobs start will auto-walk the case fine once the start event exists.
 
 Once it actually starts walking, watch your Serverless consumption tick up when it hits the first agent task, that part of your config is fine. If the start event is present and it

If your start event has no bpmn:extensionElements / uipath:entryPointId , the engine has no trigger to attach to → it provisions the job shell (RuntimeType CaseManagement, no host) and parks forever. The “0 elements / StartedTimeUtc 0001-01-01 / 0 unattended sessions” combo is exactly what an unbound entry point looks like.
 
 That entryPointId GUID must be registered in entry-points.json. The CLI frequently emits an empty/partial entry-points file. If the start event references an entry point that isn’t in entry-points.json (or it’s empty), same result, nothing to bind. Make the GUID in the start event and the one registered match.
 
 The honest gotcha: the runtime-valid compiled .bpmn really only comes out of the Studio Web canvas. This is the one that bit me hardest. uip … pack does not truly compile your caseplan.json into a runtime-executable definition… it’ll happily carry whatever .bpmn is in the folder, but a hand/CLI-produced .bpmn can be structurally present and still not be a definition the engine will walk. We never got a CLI-only case to execute; the thing that made it run was importing the solution into Studio Web once, opening the case canvas, and Publishing (that’s what compiles the .bpmn and registers the entry-point trigger), then deploying that artifact. Ironically the canvas’s bad habit is dropping the start event on round-trip, so if you do this, re-verify the start event survived the publish.
 
 Root metadata flags. Make sure the case root has publishVersion: 2, intsvcActivityConfig: “v2”, caseAppEnabled: true (with a caseAppConfig), and the unified-schema flag set. Missing these has blocked clean provisioning for us. Also: the case project must be flat, not nested under a content/ dir.

--- POST 100 @ForsakenFiji 2026-06-27T01:11:45.513Z ---
I fixed it in my project. I was getting the same error. It seems that the latest CLI has issues.
 Steps I did.
 
 Confirm it’s the format bug. Unzip your published/packed solution and check packageInfo.json:
 
 {“version”:“v1”,“type”:“unified”} + a resources/ directory = unified (rejected).
 {“version”:“v1”} (no type) + a single resources.json file + configurations/default/configuration.json + files//*.nupkg = legacy (accepted).
 
 
 Get a legacy-format baseline. Download the last version you successfully published, it’s already legacy:
uip solution packages download -d ./legacy
 If you’ve never published successfully, you’ll need a legacy package from any working publish to use as the template.
 Build the project(s) you changed with normal tooling and extract just that project’s .nupkg (the whole-solution pack being unified doesn’t matter, you only need the inner per-project nupkg).
 Preserve the entry-point id. Re-packing regenerates the entry-point uniqueId in content/entry-points.json, which breaks the upgrade with Cannot set entry point … not found in version X. So build your new nupkg from the old project nupkg: replace only the content files you actually changed, keep the old content/entry-points.json, and bump the version in the .nuspec and the .psmdcp core-properties.
 Splice into the legacy package. Drop your corrected nupkg into the same files// folder (reuse the guid), then bump that one project’s version in:
 
 
 resources.json — package key, and the file entry’s name, url, and version (the easy miss → “Required file dependency not found”),
 configurations/default/configuration.json — fileName, resourceKey, package.key, packageVersion,
 solutionMetadata.json — packageVersion + a fresh packageVersionKey GUID.
Leave packageInfo.json as {“version”:“v1”}. Verify: no occurrence of the old version string anywhere near your project name.
 
 
 Publish the spliced legacy package.

--- POST 101 @ForsakenFiji 2026-06-27T01:29:30.581Z ---
Here’s a friendly version - I asked Claude code to write a easy step-by-step guide:
 
 We ran into this identical error (“No solution tool factory is registered”) and eventually root-caused it. It’s a server-side packaging failure in Studio Web - it usually is NOT your project being broken, so don’t panic. Work through these in order, cheapest first:
 
 
 Isolate an offending project (do this first).
Publish each member project individually from its own editor. If one fails its own publish, that single project is the culprit - fix or remove it and re-add. The solution pack fails as a whole if any one member can’t pack.
 
 If every member publishes fine on its own but the solution still fails → it’s the platform packer, not you. That’s the case we were in. Two things strongly correlated with the failure for us:
 
 A Maestro Flow (or Maestro Case) in the solution. If yours contains one, temporarily remove it, then try to publish the solution again. If it succeeds without the Maestro Flow, you’ve confirmed the trigger. (UiPath acknowledged this as a known bug on the forum and routed it to engineering, it’s a server-side pack/format issue, not your code.)
 An outdated UiPath CLI if you’re packing locally - old versions couldn’t pack enriched Maestro Flow files.
 
 
 Try publishing via the CLI instead of the Studio Web “Publish” button. The browser publish uses a server-side snapshot packer (the thing that’s throwing). The CLI packs locally and often either works or gives a much clearer error:
 npm i -g @uipath /cli@latest # get off old versions
uip login
uip solution pack # produces a .zip
uip solution publish <the.zip>
 
 If the CLI publish instead says “No root solution folder was found”, that’s the same underlying platform bug (the tenant feed rejecting the package format) - see step 4.
 
 If you’ve published a working version before, you can ship a fix without the broken packer: download your last good published package (uip solution packages download -d ./pkg), swap in only the project you changed, bump that project’s version refs, re-zip, and uip solution publish it. This bypasses the snapshot packer entirely.
 
 A few extra notes for anyone comfortable with the CLI:
 
 The two errors are the same bug, two surfaces: Studio Web publish → “No solution tool factory is registered”; CLI publish → “No root solution folder was found”. Both are the tenant solution feed rejecting the package format the current tooling emits.
 Removing the Maestro Flow did NOT fix it for us (the format is the same with or without it) - but it’s still worth testing, because for some people the Maestro Flow is what makes the local pack fail, which is a different, fixable problem from the feed rejection.
 The legacy-format splice (step 4) is the only thing that reliably got a fixed package onto the feed for us, but it requires a prior good published version to download and swap into. No prior version = no splice; those folks are stuck on the platform bug and should escalate.
